# Quality Loop — Continuous Quality Improvement

Test user stories, find issues, diagnose root causes, fix with review gates, verify.

## Step 1: Parse Intent

Read the user's message and extract:

- **Area/topic**: what part of the app to focus on (e.g., "feed", "admin", "campaign", "sharing")
- **Iteration count**: how many rounds (default: 5, max enforced: 3 per session)
- **Resume intent**: does the user want to continue a previous run?

Resume indicators: "continue", "resume", "pick up", "where we left off"

If iteration count > 3, acknowledge it but explain: "Will run 3 this session (context budget). Resume for the rest."

## Step 2: Check for Resume

If resume intent detected:

1. Check if `tmp/quality-loop/state.json` exists
2. Read it — get branch name, completed iterations, next story
3. **Lock check**: if `tmp/quality-loop/state.lock` exists but state has `stoppedReason` set, the lock is stale — remove it. If state has no stoppedReason, warn user another run may be active.
4. Verify branch exists: `git branch --list {branch}`
5. Verify working tree is clean: `git status --porcelain`
6. If clean, check out the branch: `git checkout {branch}`
7. If not clean, ask user to commit or stash first
8. Skip to Step 6 (begin iteration) with the next unprocessed story

If no state file: tell user "No previous run found. Starting fresh."

## Step 3: Pre-flight Checks

Before starting, verify everything works:

1. **Dev servers**: Run `curl -s -o /dev/null -w "%{http_code}" http://ecomitram.localhost:1355` — expect 200. Run `curl -s -o /dev/null -w "%{http_code}" http://api.ecomitram.localhost:1355/health` — expect 200. If either fails, tell user to start servers and STOP.

2. **Lock check**: If `tmp/quality-loop/state.lock` exists, warn: "A quality-loop run may be active. Remove the lock file or wait." STOP.

3. **Working tree**: `git status --porcelain` — if not empty, ask user to commit or stash. STOP.

4. **MCP check**: Spawn a subagent that navigates to `http://ecomitram.localhost:1355` via Playwright MCP and returns the page title. If the subagent fails or returns no title within 30 seconds, MCP is unavailable — tell user and STOP.

5. **GitHub labels**: Ensure required labels exist (create if missing):
   ```bash
   gh label create quality-loop --description "Auto-generated by quality loop" --color "0E8A16" 2>/dev/null || true
   gh label create needs-review --description "Needs human review" --color "D93F0B" 2>/dev/null || true
   gh label create summary --description "Quality loop run summary" --color "0075CA" 2>/dev/null || true
   ```

## Step 4: Discover Stories for Area

1. Glob all files: `docs/user-stories/*.md`
2. For each file, Read the first 20 lines to get the title and story IDs
3. Fuzzy match the user's area keyword against:
   - Filename (e.g., `02-feed.md` matches "feed")
   - Title/heading text
   - Keywords in acceptance criteria (scan first 50 lines)
4. Rank: filename match (score 3) > title match (score 2) > content match (score 1)
5. Collect all stories with score >= 1, sorted by score then filename

Present to user:
```
Matched stories for "feed":
  02-feed.md (filename match)
  03-post-detail.md (content match)
  04-create-post.md (content match)
  47-feed-share.md (filename match)

Will run 3 iterations this session.
```

If no matches: list all available story filenames and ask user to refine.
If too many (>15): ask user to narrow the area.

## Step 5: Initialize Run

1. Generate timestamp: `date +%Y-%m-%d-%H%M`
2. Create branch: `git checkout -b quality-loop/{area}-{timestamp}`
3. Create state directory: `mkdir -p tmp/quality-loop`
4. Verify `tmp/` is in `.gitignore` — if not, add it (state files should not be committed)
5. Write lock file: `echo $$ > tmp/quality-loop/state.lock`
6. Write initial state.json:

```json
{
  "runId": "{area}-{timestamp}",
  "area": "{area}",
  "branch": "quality-loop/{area}-{timestamp}",
  "totalIterations": "{user_requested_count}",
  "completedIterations": 0,
  "sessionBudget": 3,
  "stories": [],
  "stoppedReason": null,
  "githubIssues": [],
  "summaryIssue": null
}
```

## Step 6: Begin Iteration

For each iteration (up to session budget of 3):

### Check context budget

If this is iteration 4+, go to Step 7 (End of Run) with reason "context-budget".

**Compaction heuristic:** At the start of each iteration, verify you can recall the branch name and run ID without reading state.json. If you cannot, context may be degraded — stop immediately and save state. This is best-effort; Claude Code doesn't expose a context usage API.

### Pick next story

From the matched stories list, pick the next one not yet processed. On second pass (cycling), skip stories with status "clean" or "skipped-all".

### Phase 1: DISCOVER

Invoke the `/test-stories` skill via the Skill tool with the current story filename as argument.

Wait for completion. Store the report text. The report is markdown with icons: `✓` = PASS, `✗` = FAIL, `⊘` = SKIP, `⚠` = Warning. Extract failed criteria (lines with `✗`) as the issue list for downstream phases.

**Gate G1:** Spawn a general-purpose subagent with AGENT-GATE.md instructions:

```
Gate: G1 (Discover → Triage)
Phase output: {test-stories report}
Issue context: Story {story-id}
Retry count: {N}
```

If SKIP: record story as "untested" in state (infra failure, not "clean"), move to next iteration.
If RETRY: re-run /test-stories once.
If PROCEED: continue to Phase 2.

### Phase 2: TRIAGE

Determine which experts to dispatch based on the discovery report:

- **Always:** ux-reviewer, code-standards
- **If UI/visual issues mentioned:** visual-reviewer
- **If mobile/responsive issues:** mobile-reviewer
- **If content/label/error message issues:** content-reviewer
- **If slow/performance issues:** performance
- **If security/auth issues:** security
- **If error/crash issues:** error-handling

Spawn each expert as a subagent (can be parallel):

```
You are reviewing issues found in story {story-id} for EcoMitram.

{expert agent instructions from ~/.claude/agents/review/{agent}.md}

## Discovery Report

{test-stories report}

## Story Content

{story file content}

Review the findings and produce your expert assessment.
```

After all experts return, spawn the triage validator subagent with AGENT-TRIAGE.md:

```
{AGENT-TRIAGE.md instructions}

## Story Reference
{story-id}: {story title}

## Discovery Report
{test-stories report}

## Expert Reviews
{all expert review outputs}

Synthesize into a single prioritized issue list.
```

**Gate G2:** Spawn gate validator with the triage output.

If SKIP (no issues survived): record story as "clean", move to next iteration.
If RETRY: re-run triage validator.
If PROCEED: continue to Phase 3.

### Phase 3: DIAGNOSE

**Issue cap:** Process at most **3 issues per iteration** (highest severity first). Remaining issues become GitHub issues for the next run. This keeps subagent count within context budget (~10 subagents per issue × 3 issues = 30 subagents max per iteration).

**Retry tracking:** Maintain an `issueRetryCount` map in memory. Every RETRY decision at any gate increments the count for that issue. Before any phase, check: if count >= 3, skip to GitHub issue.

For each issue (up to 3, highest severity first):

Spawn TWO hypothesis investigators as subagents (in parallel):

**Investigator A (Frontend):**

```
{AGENT-DIAGNOSE.md instructions}

## Your Perspective: FRONTEND

## Issue to Investigate
{issue from triage list}

## Project Root
/path/to/your/project

Investigate this issue from the frontend perspective.
```

**Investigator B (Backend):**

```
{AGENT-DIAGNOSE.md instructions}

## Your Perspective: BACKEND

## Issue to Investigate
{issue from triage list}

## Project Root
/path/to/your/project

Investigate this issue from the backend perspective.
```

After both return, spawn the diagnosis validator with AGENT-DIAGNOSE-VALIDATOR.md:

```
{AGENT-DIAGNOSE-VALIDATOR.md instructions}

## Issue
{issue from triage list}

## Frontend Investigation
{Investigator A output}

## Backend Investigation
{Investigator B output}

Compare findings and confirm root cause.
```

**Gate G3:** Spawn gate validator with the diagnosis validator output.

If SKIP: create GitHub issue for this issue, continue to next issue.
If RETRY: re-run investigators with additional context.
If PROCEED: continue to Phase 4 for this issue.

### Phase 4: FIX

Spawn implementation subagent with AGENT-FIX.md:

```
{AGENT-FIX.md instructions}

## Confirmed Diagnosis
{diagnosis validator output}

## Story Reference
{story-id}: {story title}

## Branch
You are on branch: {branch name}
Commit your fix to this branch.

## Project Root
/path/to/your/project

Implement the fix now.
```

After implementation returns, record the commit hash. Then spawn TWO review subagents (in parallel):

**Code Reviewer** (uses existing code-standards agent):

```
{instructions from ~/.claude/agents/review/code-standards.md}

## Context

You are reviewing a fix made by another agent for the quality-loop.
The fix was for: {issue description}
Root cause was: {diagnosis}

Run: git diff HEAD~1
Project: /path/to/your/project

In addition to your normal review, specifically check:
- Does it actually fix the stated root cause? (not a symptom fix)
- Does it change only what's necessary? (no scope creep)

Return: APPROVE or REJECT with specific reasons.
```

**Product Expert** (uses ux-reviewer agent):

```
{instructions from ~/.claude/agents/review/ux-reviewer.md}

## Context

You are reviewing a fix from the product/UX perspective.
The fix was for: {issue description}
Root cause was: {diagnosis}

Run: git diff HEAD~1
Project: /path/to/your/project

Evaluate:
- Does this fix actually improve the user experience?
- Could this make something else worse for the user?
- Is the fix complete, or is it a partial/symptomatic fix?

Return: APPROVE or REJECT with specific reasons.
```

**Gate G4:** Both must APPROVE. If either REJECT:

- If retry budget allows: spawn fix agent again with reviewer feedback
- If retry budget exhausted: revert all commits from this story's issues (`git revert {commit-hash} --no-edit` for each tracked commit), create GitHub issue, continue to next issue

### Phase 5: VERIFY

Invoke `/test-stories` skill again with the same story filename.

**Gate G5:** Spawn gate validator.

- If test passes: PROCEED to report
- If test fails and looks like a flake: RETRY once
- If test fails consistently: revert all commits from this iteration (`git revert {commit-hash} --no-edit` for each), create GitHub issue

### Phase 6: REPORT

Update state.json with this iteration's results:

```json
{
  "id": "{story-id}",
  "status": "fixed | clean | untested | skipped-all | partial",
  "issuesFound": 0,
  "issuesFixed": 0,
  "issuesSkipped": 0,
  "issuesDeferred": 0,
  "commits": ["abc1234"],
  "issueAttempts": { "issue-title": 2 }
}
```

Log to terminal:

```
Iteration {N}/{total}: {story-id}
  Issues found: X | Fixed: Y | Skipped: Z
  Commits: {list}
```

Move to next iteration.

## Step 7: End of Run

When all iterations complete OR session budget exhausted:

1. **Update state:**
   - Set `stoppedReason`: "completed" | "context-budget" | "no-more-stories"
   - Set `completedIterations`

2. **Regression sweep** (only if fewer than 3 iterations ran, leaving context budget):
   Run `/test-stories` on 1-2 adjacent stories in the same area that weren't tested this run.
   If any fail, log as regression warnings in the summary issue. Do not attempt to fix — just report.

3. **Push branch to remote:**

   ```bash
   git push -u origin {branch}
   ```

4. **Create GitHub issues for skipped items:**
   For each issue marked as skipped/needs-review during the run:

   ```bash
   gh issue create \
     --title "[quality-loop] {story-id}: {issue summary}" \
     --label "quality-loop,needs-review,{area}" \
     --body "..."
   ```

   Store issue numbers in state.

5. **Create summary GitHub issue:**

   ```bash
   gh issue create \
     --title "[quality-loop] Run summary: {area} ({date})" \
     --label "quality-loop,summary" \
     --body "..."
   ```

   Body includes:
   - Branch link
   - Iterations completed / total
   - Per-story results table
   - Links to all commits
   - Links to all skipped-item issues
   - Stats: issues found, fixed, skipped

6. **Remove lock file:**

   ```bash
   rm -f tmp/quality-loop/state.lock
   ```

7. **Print final summary:**

   ```
   Quality Loop Complete: {area}
   Branch: quality-loop/{area}-{timestamp}
   Iterations: {completed}/{total}
   Issues: {found} found, {fixed} fixed, {skipped} skipped
   Summary: {github issue URL}

   {if stopped early:}
   Resume with: "continue quality loop"
   ```

## Rules

- Do NOT open browser yourself — /test-stories and subagents handle all browser interaction
- Do NOT skip phases — every phase must run, every gate must validate
- Do NOT proceed after context compaction — stop cleanly
- Do NOT fix symptoms — Phase 3 exists to find root causes
- Do NOT invent problems — "no issues" is a valid, positive outcome
- Keep main context clean — store only summary text from subagents, not full outputs
- Sequential stories, never parallel — one iteration at a time
- Every fix must be cross-checked by agents that didn't write it
